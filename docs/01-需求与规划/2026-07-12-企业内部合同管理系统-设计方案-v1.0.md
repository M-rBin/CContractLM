# 企业内部合同管理系统设计方案

## 一、项目概述

企业内部合同管理系统面向企业内部员工、业务负责人、财务人员、法务人员和系统管理员，建设目标是将企业日常合同从分散文件、线下表格和人工提醒中集中到统一系统中管理。

第一版系统定位为“合同工作台 + 合同台账 + 附件归档 + 履约跟踪 + 收付款计划 + 站内提醒 + 经营分析”。系统暂不建设线上审批流、合同在线编辑、电子签章、复杂权限隔离和财务系统对接，优先解决合同签署后的集中查询、执行跟踪、到期提醒和收付款跟进问题。

系统基于当前项目已有的管理端与服务端能力扩展，复用用户、角色、部门、菜单、权限、日志等基础模块。合同数据第一版对所有登录用户可见，新增、编辑、删除、导出、附件上传等操作仍通过角色权限控制。

### 1.1 建设目标

- 建立统一的合同台账，减少合同信息散落在个人电脑、聊天记录和表格中的情况。
- 支持合同正文、补充协议、验收材料、发票、付款凭证等多类型附件归档。
- 跟踪合同履约节点，减少交付、验收、续签、终止等事项遗漏。
- 跟踪收付款计划，形成待收、待付、已收、已付、逾期等状态管理。
- 通过站内提醒提示合同到期、节点临近、收付款到期和逾期事项。
- 提供登录后默认工作台，帮助用户快速总览合同关键指标、待办提醒和经营看板。
- 提供经营分析视图，辅助管理者了解合同规模、金额分布、收付款进度和逾期风险。

## 二、功能范围

第一版系统包含以下核心功能模块。

### 2.1 合同工作台

合同工作台作为用户登录后的默认首页，用于总览当前合同关键指标、待办提醒、看板图表、快捷入口和最近动态。

关键指标包括合同总数、履行中合同、即将到期合同、逾期合同、待收金额、待付金额和逾期金额。待办提醒包括合同到期提醒、履约节点提醒、收款计划提醒和付款计划提醒。看板图表包括合同状态分布、合同类型分布、月度签约金额和收付款完成情况。

### 2.2 合同台账

合同台账是合同管理的主入口，支持合同新增、编辑、查看、删除、查询、筛选和导出。合同基础信息包括合同编号、合同名称、合同类型、签约主体、相对方、合同金额、币种、签订日期、生效日期、到期日期、负责人、合同状态和备注。

合同类型默认包括采购合同、销售合同、服务合同、租赁合同、框架协议和其他合同。合同状态默认包括草稿、履行中、已到期、已终止和已归档。

### 2.3 合同附件

合同附件支持按照业务材料类型分类上传、查看、下载和删除。附件分类包括合同正文、补充协议、验收材料、发票、付款凭证和其他附件。

第一版附件管理只做分类归档，不做附件版本追踪和在线预览强依赖。

### 2.4 履约节点

每份合同可维护多个履约节点。节点可用于记录交付、验收、续签、到期、终止等关键事项。每个节点包含节点名称、计划日期、实际完成日期、负责人、状态和说明。

### 2.5 收付款计划

每份合同可维护多期收款或付款计划。计划内容包括收付方向、计划金额、计划日期、实际金额、实际日期、状态和备注。状态包括待处理、已完成和逾期。

### 2.6 站内提醒

系统根据合同到期日、履约节点计划日期、收付款计划日期生成站内提醒。提醒集中展示在待办或提醒列表中，用户可从提醒跳转到来源合同、履约节点或收付款计划。

### 2.7 经营分析

经营分析用于展示合同整体情况和履约收付款风险，包括合同总数、合同总金额、按类型统计、按状态统计、按月份统计、应收应付、已收已付、待收待付、逾期金额、即将到期合同数量和逾期履约节点数量。

## 三、核心模块设计

### 3.1 菜单与页面结构

用户登录后默认进入合同工作台。系统在管理端新增“合同管理”一级菜单，下设以下页面：

- 合同台账：合同查询、新增、编辑、查看、删除和导出入口。
- 合同详情：展示合同基础信息、附件、履约节点、收付款计划和操作记录。
- 履约提醒：集中展示即将到期、已逾期、待完成的合同履约节点。
- 收付款管理：按收款计划和付款计划维度查询、处理和统计。
- 经营分析：展示合同经营指标、金额分布、收付款进度和逾期风险。

### 3.2 合同工作台交互

合同工作台是登录后的默认首页，展示用户需要优先关注的合同概况和待办事项。页面顶部展示关键指标，包括合同总数、履行中合同、即将到期合同、逾期合同、待收金额、待付金额和逾期金额。

工作台中部展示待办提醒和看板图表。待办提醒按合同到期、履约节点、收款计划和付款计划分类展示，用户可点击提醒跳转到对应合同详情或处理页面。看板图表展示合同状态分布、合同类型分布、月度签约金额和收付款完成情况。

工作台提供快捷入口，包括新增合同、合同台账、履约提醒和收付款管理。页面底部展示最近动态，包括最近新增合同、最近完成履约节点和最近完成收付款记录。

### 3.3 合同台账交互

合同台账默认以列表展示合同。列表支持按合同名称、合同编号、合同类型、合同状态、相对方、负责人、签订日期、生效日期、到期日期等条件筛选。

新增或编辑合同时，用户填写合同基础信息。保存后，系统生成合同记录，用户可继续进入详情页维护附件、履约节点和收付款计划。

合同详情采用分区或页签展示，建议包含“基础信息”“附件”“履约节点”“收付款计划”“操作记录”。用户从同一合同详情内完成相关信息维护，减少在多个页面之间反复跳转。

### 3.4 履约节点交互

用户在合同详情中新增履约节点，填写节点名称、计划日期、负责人和说明。节点未完成且计划日期临近时，系统生成站内提醒；节点超过计划日期仍未完成时，标记为逾期。

负责人或有权限人员可将节点标记为已完成，并填写实际完成日期和完成说明。

### 3.5 收付款计划交互

用户在合同详情中新增收付款计划，选择收付方向并填写计划金额、计划日期和备注。系统根据计划日期判断待处理、临近到期和逾期状态。

当实际收款或付款发生后，用户填写实际金额和实际日期，系统将计划状态更新为已完成。若只完成部分金额，第一版可通过备注说明，不做复杂拆分；后续可扩展为分批核销。

### 3.6 站内提醒交互

站内提醒列表展示提醒类型、来源合同、提醒内容、计划日期、逾期状态和处理状态。用户点击提醒后可跳转到合同详情、履约节点或收付款计划。

提醒类型包括合同到期提醒、履约节点提醒、收款计划提醒、付款计划提醒和逾期提醒。

### 3.7 经营分析交互

经营分析页以指标卡、图表和明细入口组成。指标卡展示合同总数、合同总金额、应收金额、应付金额、待收金额、待付金额和逾期金额。

图表展示合同类型分布、合同状态分布、月度签约金额趋势、收付款计划完成情况。用户可按日期范围、合同类型和合同状态筛选分析结果。

## 四、数据结构与流向

### 4.1 核心数据实体

合同主表保存合同基础信息，是附件、履约节点、收付款计划和提醒的主关联对象。

合同附件表保存附件分类、文件名称、文件地址、文件大小、上传人、上传时间和所属合同。

履约节点表保存节点名称、计划日期、实际完成日期、负责人、状态、说明和所属合同。

收付款计划表保存收付方向、计划金额、计划日期、实际金额、实际日期、状态、备注和所属合同。

提醒表保存提醒类型、来源对象、来源合同、提醒内容、提醒时间、处理状态和处理时间。

操作记录表保存合同相关关键操作，包括新增、编辑、删除、附件上传、节点完成、收付款完成等。

工作台数据由合同主表、履约节点表、收付款计划表、提醒表和操作记录表汇总生成，不单独维护重复业务数据。

### 4.2 主要数据流

合同登记流程：用户新增合同基础信息，系统保存合同主表记录，合同进入草稿或履行中状态。用户随后补充附件、履约节点和收付款计划。

附件归档流程：用户在合同详情上传附件并选择附件分类，系统保存文件并记录附件元数据。附件与合同建立关联，后续可按合同或附件类型查询。

履约跟踪流程：用户维护履约节点，系统根据计划日期判断节点是否临近或逾期。节点完成后，用户录入实际完成日期，系统更新节点状态。

收付款跟踪流程：用户维护收付款计划，系统根据计划日期和完成情况判断待处理、已完成或逾期。实际收付款发生后，用户录入实际金额和实际日期。

提醒生成流程：系统定期扫描合同到期日、履约节点计划日期和收付款计划日期，生成或更新站内提醒。用户处理相关事项后，提醒状态同步更新。

经营分析流程：系统基于合同主表、履约节点和收付款计划汇总统计指标，为经营分析页面提供数据。

工作台展示流程：用户登录后进入合同工作台，系统汇总合同主表、履约节点、收付款计划、提醒和操作记录，展示关键指标、待办提醒、看板图表、快捷入口和最近动态。

## 五、边界情况处理

### 5.1 权限边界

第一版合同数据对所有登录用户可见，不按部门、负责人或业务线隔离。合同新增、编辑、删除、导出、附件上传、附件删除、履约节点维护、收付款计划维护等操作通过角色权限控制。

无权限用户可查看合同信息，但不能执行受控操作。

### 5.2 审批边界

第一版不建设线上审批流。合同是否已经完成内部审批，由业务人员在线下处理后在系统中登记。系统仅保存合同状态和相关附件，不判断审批链路是否完整。

### 5.3 附件边界

附件支持分类归档，但不做历史版本管理。若同一合同存在多个版本文件，用户可通过附件名称、附件分类和备注区分。

附件上传失败时，系统不创建附件记录；附件记录删除时，应保留合同主数据和其他业务数据。

### 5.4 日期与逾期边界

合同到期、履约节点逾期、收付款逾期均基于计划日期和完成状态判断。已完成事项不再按计划日期标记为逾期。

合同到期不自动等同于合同终止。到期状态可由系统提示，最终合同状态由有权限人员确认维护。

### 5.5 金额边界

合同金额、计划金额和实际金额第一版只做记录和统计，不做财务凭证、发票核销、银行流水匹配和复杂汇率换算。

若合同存在部分收付款，第一版通过实际金额和备注记录，不拆分为多次核销明细。需要精细核销时，可在后续版本扩展。

### 5.6 提醒边界

第一版只做站内提醒，不接入邮件、短信、企业微信、钉钉等外部通知渠道。

提醒用于辅助用户跟进，不替代业务责任判断。提醒生成失败不应影响合同主数据、履约节点和收付款计划的正常维护。

## 六、后续规划

第一版完成后，可根据实际使用情况逐步扩展以下能力：

- 线上审批流：按合同类型、金额、部门配置不同审批路径。
- 附件版本管理：记录合同正文和补充协议的历史版本、变更说明和上传人。
- 在线预览：支持 PDF、Word、图片等合同附件在线预览。
- 外部通知：接入邮件、企业微信、钉钉或短信提醒。
- 财务对接：与财务系统、发票系统或银行流水系统对接，实现更精细的收付款核销。
- 权限细化：按部门、负责人、合同类型或授权范围控制合同可见性。
- 合同风险分析：基于合同字段和履约状态识别到期风险、逾期风险和金额风险。

## 七、结论

本方案建议第一版采用“工作台 + 台账 + 履约 + 收付款”的建设路径。该路径能在不引入复杂审批和外部系统对接的前提下，优先解决企业合同管理中最常见的总览看板、集中归档、执行跟踪、到期提醒和经营统计问题，适合作为企业内部合同管理系统的首个可交付版本。
