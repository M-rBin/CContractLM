# 需求背景与目标

## 需求背景

系统设置模块是企业内部合同管理系统的权限与基础数据底座，为合同管理提供账号、组织和权限支撑。当前存在以下痛点：

1. **账号与权限管理分散**：合同管理涉及业务负责人、财务、法务等多类角色，若账号和权限缺乏统一维护入口，容易出现权限过大或越权操作。
2. **组织与岗位数据缺失基准**：合同负责人、履约节点负责人均需从企业组织中选取，缺少统一的部门和岗位数据会导致人员归属混乱、选择困难。
3. **菜单与权限点缺乏可视化配置**：功能菜单和操作按钮的权限若硬编码，角色权限调整需改动代码，无法灵活配置。
4. **枚举数据硬编码难维护**：合同类型、附件分类等枚举若硬编码在系统中，业务调整时需改动代码，无法由管理员自助维护（字典管理，本期待实现）。

## 产品目标

1. **账号统一管理**：集中维护登录账号及其部门、岗位、角色归属，支持按姓名/账号/手机号检索和批量维护。
2. **权限精确控制**：基于角色的访问控制（RBAC），角色通过勾选菜单树（目录/菜单/按钮）获得功能与操作权限，权限配置可视化。
3. **组织数据规范**：以层级结构维护部门（支持省公司/分公司/部门分类），组织数据作为用户归属和合同负责人选择的来源。
4. **菜单可配置**：目录、菜单、按钮三级可视化维护，支持权限标识、路由、图标、显示隐藏配置。
5. **基础数据自助维护**：字典类型与字典项支持管理员自助增删改（本期待实现）。

## 产品边界

| 纳入范围 | 排除范围（后续迭代规划）|
|---|---|
| 1. 用户管理（增删改查、批量删除、启停、部门树筛选） | 1. 用户自助注册、找回密码、重置密码 |
| 2. 部门管理（层级 + 部门类型） | 2. 组织架构导入导出、批量调岗 |
| 3. 岗位管理 | 3. 岗位与薪酬、绩效等 HR 能力联动 |
| 4. 角色管理与菜单权限分配 | 4. 数据行级权限、字段级权限隔离 |
| 5. 菜单管理（目录/菜单/按钮三级） | 5. 用户导入/导出、批量设置岗位/角色（界面已预留，功能后续实现） |
| 6. 字典管理（待实现） | 6. 多语言字典、字典版本管理 |
| 7. 操作日志（待实现） | 7. 日志告警、行为分析、外部审计对接 |

> **说明**：系统设置模块复用企业已有管理平台基础能力，字段与交互沿用平台既有规范。菜单结构为「组织管理」（部门/用户/岗位）与「权限管理」（角色/菜单）两个一级菜单。合同业务数据对所有登录用户可见，本模块权限控制的是各模块的受控操作。

### US-001：用户账号管理

**用户故事**
> 作为 **系统管理员**，
> 我想要 **新增、编辑、删除、批量删除用户账号并为其分配部门、岗位和角色，同时可启用或停用账号**，
> 以便 **企业员工能按各自角色登录并使用系统对应功能**。

**优先级**：P0（MVP）　　**估算**：3 人天　　**依赖**：US-002、US-003、US-004

**验收标准：**
- [ ] AC1：填写姓名、账号、密码、部门、岗位、角色并提交，新增成功后列表出现该用户，提示"新增成功"。
- [ ] AC2：编辑时账号不可修改（禁用输入），不需填写密码，其余字段可改，提交提示"更新成功"。
- [ ] AC3：可按姓名/账号/手机号关键字与员工状态筛选，可点击左侧部门树按部门过滤用户。
- [ ] AC4：切换列表中的状态开关，账号在启用/禁用间切换，提示"状态更新成功"，禁用后不能登录。
- [ ] AC5：勾选多个用户后可批量删除，删除前二次确认；单条删除同样二次确认，提示"删除成功"。
- [ ] AC6：列表手机号脱敏展示。

> **说明**：导入、导出、批量设置岗位、批量设置角色的入口已在界面预留，当前提示"功能开发中"，不在本期验收范围。用户账号无「重置密码」功能。

**关联功能需求**：见 3.1 用户管理　　**关联数据模型**：sys_user、sys_user_role

# 功能需求详述

> 系统设置分为「组织管理」（部门管理、用户管理、岗位管理）与「权限管理」（角色管理、菜单管理）两个一级菜单，仅系统管理员可维护。以下接口路径、字段名与 admin 前端已实现代码对齐（接口前缀 `/admin/sys/`）。字典管理、操作日志前端待实现，作占位说明。

## 用户管理（组织管理）

> **模块职责**：维护系统登录账号及其所属部门、岗位、角色，是权限控制的账号基础。页面采用左侧部门树 + 右侧用户列表布局。

### 功能说明

- 列表查询：分页查看用户，支持按关键字（姓名/账号/手机号）、员工状态筛选，点击左侧部门树按部门过滤，列表手机号脱敏展示
- 新增用户：维护姓名、账号、密码、手机号、所属部门、岗位、角色、邮箱
- 编辑用户：修改用户信息，账号不可修改，无需填写密码
- 删除用户：单条删除，二次确认
- 批量删除：勾选多条后批量删除，二次确认
- 启用/停用：列表状态开关直接切换
- 导入、导出、批量设置岗位、批量设置角色：入口已预留，当前提示"功能开发中"，本期不实现

### 接口规格（API / 消息 / 事件）

#### 分页查询用户

**GET /admin/sys/user/list**

**Request:**
```json
{
  "keyword": "string，姓名/账号/手机号关键字，可选",
  "status": "int，状态：1启用/0禁用，可选",
  "departmentId": "int，部门ID，来自左侧部门树点击，可选",
  "page": "int，页码，从1开始",
  "pageSize": "int，每页条数，默认10"
}
```

**Response (200):**
```json
{
  "code": 200,
  "data": {
    "list": [
      {
        "id": "int，用户ID",
        "name": "string，姓名",
        "username": "string，账号",
        "department": "{ id, name }，所属部门",
        "position": "{ id, name }，岗位",
        "userRoles": "[{ role: { id, name } }]，角色列表",
        "phone": "string，手机号（前端脱敏展示）",
        "status": "int，状态：1启用/0禁用"
      }
    ],
    "pagination": { "page": "int", "pageSize": "int", "total": "long" }
  }
}
```

**错误码：**

| 错误码 | 含义 | 处理建议 |
|--------|------|---------|
| 401 | 未登录或凭证失效 | 跳转登录页 |
| 403 | 无权限 | 提示无权限并隐藏入口 |

**幂等性**：查询接口，天然幂等。

#### 新增用户

**POST /admin/sys/user/add**

**Request:**
```json
{
  "username": "string，账号，必填",
  "password": "string，密码，必填",
  "name": "string，姓名，必填",
  "phone": "string，手机号，可选",
  "email": "string，邮箱，可选（填写时校验格式）",
  "departmentId": "int，所属部门ID，必填",
  "positionId": "int，岗位ID，单选，可选",
  "roleIds": "int[]，角色ID列表，可选"
}
```

**Response (200):** `{ "code": 200, "msg": "新增成功" }`

**错误码：**

| 错误码 | 含义 | 处理建议 |
|--------|------|---------|
| 40001 | 账号已存在 | 后端返回错误信息，前端由拦截器统一提示 |

**幂等性**：非幂等，前端提交时按钮 loading 防重复提交。

#### 编辑用户

**PUT /admin/sys/user/update**：请求体含 `id`，不含 `username`、`password`（账号不可改、不改密码）。返回 `{"code":200,"msg":"更新成功"}`。

#### 删除 / 批量删除用户

- **DELETE /admin/sys/user/delete/{id}**：单条删除，返回"删除成功"
- **POST /admin/sys/user/batch-delete**：`{ "ids": "int[]" }`，批量删除，返回"批量删除成功"

**幂等性**：重复删除返回已删除，幂等。

#### 启用/停用

**PUT /admin/sys/user/update-status**

**Request:** `{ "id": "int", "status": "int，1启用/0禁用" }`
**Response:** `{ "code": 200, "msg": "状态更新成功" }`

**幂等性**：以目标状态为准，幂等。

### 业务规则

| 编号 | 规则 | 触发条件 | 处理方式 |
|------|------|---------|---------|
| BR-U01 | 账号唯一 | 新增提交时 | 由后端校验唯一性，重复时返回错误信息，前端统一提示 |
| BR-U02 | 账号不可修改 | 编辑用户时 | 账号输入框禁用，不参与更新 |
| BR-U03 | 停用禁止登录 | 用户状态为禁用 | 该账号登录被拒 |

# 状态机

> 系统设置模块中具备状态流转的对象为「用户账号」（启用/禁用）与「菜单」（显示/隐藏）。角色的状态字段已维护但列表暂无开关；部门、岗位的状态字段为接口预留，当前 UI 未提供切换。

## 用户账号状态机

| 状态 | 触发动作 | 触发方 | 下一状态 | 后续操作/副作用 |
|---|---|---|---|---|
| （无） | 新增用户 | 系统管理员 | 启用 | 生成账号，默认启用（status=1），可登录 |
| 启用 | 切换状态开关 | 系统管理员 | 禁用 | 账号不能登录 |
| 禁用 | 切换状态开关 | 系统管理员 | 启用 | 账号恢复登录能力 |
| 启用/禁用 | 删除 / 批量删除 | 系统管理员 | （已删除） | 账号删除，其角色关联一并解除 |

## 菜单显示状态机

| 状态 | 触发动作 | 触发方 | 下一状态 | 后续操作/副作用 |
|---|---|---|---|---|

# 业务主流程

系统管理员搭建权限基础数据的端到端主流程：

```
新增部门（管理员，含部门类型）--> 新增岗位（管理员）
  --> 维护菜单树：目录/菜单/按钮 + 权限标识（管理员）
  --> 新增角色（管理员）--> 为角色勾选菜单树权限（管理员）
  --> 新增用户并绑定部门/岗位/角色（管理员）
  --> 用户使用账号密码登录（用户，系统校验状态与凭证）
  --> 系统按用户所属角色的菜单权限展示功能入口（系统自动）
```

# 核心业务规则

> 以下为跨模块的全局规则，各模块内的规则见对应模块的业务规则表，不在此重复。规则以 admin 前端已实现行为为准。

| 编号 | 规则 | 触发条件 | 处理方式 |
|------|------|---------|---------|
| BR-G01 | 系统设置仅管理员可维护 | 访问组织管理/权限管理各模块 | 仅系统管理员可维护，其他角色按菜单权限控制入口可见性 |
| BR-G02 | 唯一性交后端校验 | 账号等唯一字段重复 | 前端不做唯一性预校验，由后端返回错误信息，前端拦截器统一提示 |
| BR-G03 | 删除统一二次确认 | 任意删除/批量删除操作 | 弹出确认框，确认后执行，成功提示"删除成功" |
| BR-G04 | 请求错误统一提示 | 接口返回错误 | 由 HTTP 拦截器统一提示，页面不重复弹窗 |
| BR-G05 | 权限点来源于菜单 | 角色分配权限时 | 角色可勾选的权限点来源于菜单树（目录/菜单/按钮） |

## 权限判定规则

> 判定逻辑：用户 → 角色（sys_user_role）→ 菜单/按钮权限（sys_role_menu）。
>
> - 用户拥有的权限 = 其绑定的所有角色权限的并集。
> - 菜单/按钮是否展示：用户权限集合包含对应菜单ID则展示，否则隐藏。
> - 边界情况：用户未绑定任何角色时仅可见默认公共页面；禁用用户不进入权限判定，直接拒绝登录。
> - 分配权限保存时提交「勾选节点 + 半选父节点」，保证父级目录/菜单可正常展开。

## 字段与实现对齐说明

> 本 PRD 已按 admin 前端实际实现对齐，与初版（v1.0）相比的主要差异：
>
> - 用户管理无「重置密码」；岗位为单选（非多选数组）；状态用「启用/禁用」。
> - 角色唯一标识字段为 `label`（非 roleKey）；角色含启用/禁用状态。
> - 部门新增「部门类型」必填字段；部门/岗位的启停为接口预留、UI 未启用。
> - 关联删除保护（角色关联用户、部门关联下级）当前前端未做前置拦截提示，由后端保证数据一致性。
> - 新增「菜单管理」模块；「字典管理」「操作日志」前端待实现。

# 非功能需求

1. **性能需求**：列表分页查询 P99 响应时间 < 1 秒；部门树、菜单树在千级节点内加载 < 2 秒。
2. **兼容性需求**：支持 Chrome、Firefox、Safari、Edge 最新版本；适配 1920×1080、1366×768 常见桌面分辨率。
3. **安全性需求**：密码加密存储；列表手机号脱敏展示；数据传输使用加密通道；禁用与删除账号即时阻断登录能力。
4. **易用性需求**：删除操作统一二次确认；表单校验失败保留已填内容并内联提示；成功/失败均有明确反馈文案；请求错误由拦截器统一提示。
