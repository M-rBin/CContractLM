## 一、范围与目标

合同管理模块覆盖 6 个业务能力：

| 模块 | 对应用户故事 | 目标 |
|------|-------------|------|
| 合同工作台 | US-001 | 登录后总览关键指标、待办提醒、看板图表、快捷入口、最近动态 |
| 合同台账 | US-002 | 合同主记录的登记、编辑、查询、删除、导出 |
| 合同详情 | US-003/004/005 | 单合同基础信息、附件、履约节点、收付款计划、操作记录 |
| 履约提醒 | US-006 | 合同到期与履约节点提醒的集中查看、跳转、完成处理 |
| 收付款管理 | US-007 | 跨合同查询、登记、统计、导出收付款计划 |
| 经营分析 | US-008 | 合同规模、金额分布、收付款进度、逾期风险统计 |

## 三、实施策略

1. 先补齐合同详情的从属数据表和接口，因为附件、履约节点、收付款计划是提醒、工作台、经营分析的数据来源。
2. 履约提醒和收付款管理分别依赖履约节点、收付款计划，放在详情之后实施。
3. 工作台依赖合同、节点、收付款、提醒四类数据，放在核心业务闭环之后实施。
4. 经营分析以查询统计为主，依赖合同与收付款计划，可在收付款管理完成后实施。
5. 每个阶段都按“后端模型/API → 前端页面/API → 菜单/权限 → 联调验证 → 文档同步”的顺序闭环。

### 测试执行发现（2026-07-13）

走查测试用例过程中发现并修复 2 个真实缺陷，均已过 code-reviewer 审查（第二个问题的第一版修复经审查后再修复一次 HIGH 级遗漏，二轮均已通过）：

1. **附件下载接口 500 错误**（对应 DET-001）。`contract-attachment.controller.ts` 的 `download` 方法用裸 `@Res()` 接管响应但仍 `return` 了 `pipe(res)` 的返回值，被全局 `TransformInterceptor` 递归格式化时命中 Express Response 对象的循环引用，导致 500。修复：方法改为不返回值（`Promise<void>`），并补上 `stream.on('error', ...)` 处理磁盘文件缺失场景（避免未捕获异常导致进程崩溃）。
2. **CSV 导出文件内容不可用**（对应 PAY-001，影响面覆盖合同台账导出、收付款管理导出，以及既有的用户导出功能）。全局 `TransformInterceptor` 会把所有 handler 返回值统一包装成 `{code, data, message}` JSON，CSV 导出接口返回的字符串被包了一层 JSON，用户下载到的文件打开后是一段 JSON 文本而非表格。修复：拦截器新增判断，当 handler 已通过 `@Header()` 显式声明非 `application/json` 的 `Content-Type` 时跳过统一包装。

## 九、阶段完成标准

每个阶段完成时必须满足：

1. 后端构建通过：`cd server && npm run build`
2. 若改前端，前端构建通过：`cd admin && npm run build`
3. 页面所有数据接口走真实 API，不使用 Mock
4. 菜单与权限点可由启动同步或种子脚本幂等补齐
5. 关键业务规则至少通过手工或脚本验证一次
6. 更新本实施计划中的状态和备注
