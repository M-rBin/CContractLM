# 二、引言

## 2.1 编写目的

本详细设计说明书面向后端与前端开发工程师，将合同管理模块 PRD 中的功能需求、接口规格和数据模型细化为可直接实现的类设计、数据库物理表结构和 API 接口定义，作为合同业务侧编码、联调与测试的直接依据。合同业务后端尚未实现，本文在现有 NestJS + Prisma + CrudControllerFactory 框架基础上进行落地设计。

## 2.2 设计依据

- 《合同管理模块PRD（简单业务）v1.0》——功能需求、业务规则、状态机、数据模型来源
- 《企业内部合同管理系统设计方案 v1.0》——总体功能范围与边界
- 现有后端框架：NestJS 11 + Prisma 6 + `CrudControllerFactory` 通用 CRUD 能力
- 编码/数据库/后端/安全技术规范（knowledge base）

## 2.3 全局约定

> 以下为贯穿所有合同模块的通用约定，各模块章节不再重复。约定以**现有框架实际实现**为准（对齐 `base_sys_*` 表与 `CrudControllerFactory`）。

**统一响应格式：**

所有接口返回统一结构，成功与失败均由响应拦截器包装：

```ts
// 单对象 / 操作类
ApiResponse<T> = { code: number; msg: string; data?: T }
// 分页列表
PageResponse<T> = { code: number; data: { list: T[]; pagination: { page: number; pageSize: number; total: number } } }
```

- 成功 `code=200`；业务错误 `code` 取 5 位分段码（4xxxx）；鉴权失败 401，无权限 403。
- 错误信息由后端返回，前端 HTTP 拦截器统一提示，页面不重复弹窗。

**公共字段约定（对齐现有框架）：**

所有合同业务表统一包含以下字段（与 `base_sys_position` 等现有表一致）：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Int（自增主键） | 主键，`@id @default(autoincrement())` |
| tenantId | Int? | 租户 ID，多租户预留，建索引 |
| createTime | DateTime | 创建时间，`@default(now())` |
| updateTime | DateTime | 更新时间，`@updatedAt` |

> 说明：现有框架未使用 `created_by / updated_by` 审计字段，合同业务的"操作人"通过独立的 `contract_oper_record` 操作记录表承载，故公共字段不引入 `created_by / updated_by`，与既有表保持一致。

**命名规范：**

- Prisma model 用 PascalCase（如 `Contract`），字段 camelCase；物理表名 snake_case 加 `contract_` 前缀（`@@map`），物理字段 snake_case。
- 接口路径前缀 `admin/contract/`，走 admin 鉴权体系；RESTful 风格。
- 索引名 `idx_表_字段`，外键字段必须建索引；低区分度字段（status、type）不单独建索引。

**错误码规范：**

| 分段 | 含义 |
|------|------|
| 200 | 成功 |
| 401 / 403 | 未登录 / 无权限 |
| 40001 | 合同编号已存在 |
| 40002 | 日期顺序非法（生效早于签订 / 到期早于生效） |
| 40003 | 关联的合同不存在 |
| 40004 | 附件超出大小限制（>50MB） |
| 40005 | 状态非法流转 |

**鉴权约定：**

- 采用 JWT + 权限码（`@Perms`）机制，复用系统设置模块的 RBAC 体系。
- 合同数据对所有登录用户可见（查询类接口仅需登录）；新增、编辑、删除、导出、上传、登记等写操作标注权限码 `contract:<资源>:<操作>`。

**通用 CRUD 约定：**

标准增删改查复用 `CrudControllerFactory`，自动提供：`GET list`（分页）、`GET detail/:id`、`POST add`、`PUT update`、`PUT update-status`、`DELETE delete/:id`、`POST batch-delete`。各模块仅对涉及事务、级联、跨表聚合的接口做自定义覆盖。

# 三、模块详细设计

> 每个功能模块含：模块职责 + 核心类/服务设计 + 核心业务流程 + 状态机（如有）+ 异常处理。标准增删改查复用 `CrudControllerFactory`，仅对涉及事务、级联、跨表聚合的接口自定义覆盖。
>
> **实现现状标注**：合同台账（`ContractInfo` model + controller/service）后端已有一版实现（路由前缀 `admin/contract/info`），本设计以 PRD 为目标（路由 `admin/contract/ledger`、金额 `Decimal(18,2)`、业务错误码 40001/40002），标注为 **[需改造]** 的部分表示现有代码需按本设计调整。其余五模块（合同详情四张从属表、履约提醒、收付款管理、工作台、经营分析）后端尚未实现，为全新设计。

## 3.1 合同台账 详细设计

**模块职责：** 合同管理主入口，提供合同主记录的分页查询、新增、编辑、详情查看、删除、导出。删除时级联清理附件、履约节点、收付款计划、相关提醒，操作记录保留。（对应 PRD US-002、BR-L01~L06）

**核心类/服务设计：**

| 类/服务名 | 职责 | 关键方法 | 复用/自定义 |
|----------|------|---------|-----------|
| ContractLedgerController（继承 `CrudControllerFactory(ContractVo)`，`@CrudController({prefix:'admin/contract/ledger'})`） | 路由与统一响应 | `list`/`detail`（复用工厂）、`add`/`update`/`delete`/`export`（自定义） | [需改造] 前缀 info→ledger |
| ContractLedgerService（继承 `BaseService`，`super(prisma,'contractInfo')`） | 业务校验、事务级联删除 | `pageContracts`/`createContract`/`updateContract`/`deleteContractCascade`/`listForExport`/`validateBusiness` | 部分自定义 |

**核心业务流程 — 新增合同：**

处理步骤：
1. Controller 经 `AddContractDto` 校验字段类型/必填/长度/枚举（type/status/currency 用 `@IsIn`）。
2. Service.validateBusiness：日期先后校验——生效早于签订或到期早于生效，抛 40002，message="日期填写有误，请检查签订、生效、到期日期的先后顺序"（BR-L02）。
3. 编号唯一校验：`findFirst({code})` 命中抛 40001，message=`合同编号【${code}】已存在，请更换`（BR-L01）。
4. 校验通过 `prisma.contractInfo.create()`，status 缺省落"草稿"。单表写入无需事务。

**核心业务流程 — 删除合同（级联事务）：**

处理步骤：
1. 前端二次确认后调用。
2. Service.deleteContractCascade 在 `prisma.$transaction` 内按序：删附件 → 删履约节点 → 删收付款计划 → 删相关履约提醒 → 删合同主记录（BR-L05）。
3. 合同操作记录（contract_oper_record）不在删除事务内，保留（PRD 明确）。
4. 记录不存在时 Prisma 抛 P2025，由 `AllExceptionsFilter` 统一转 400「记录不存在」。

**状态机（合同状态）：**

| 当前状态 | 触发条件 | 触发方 | 流转至 | 系统动作 |
|---------|---------|-------|-------|---------|
| （无） | 新增合同 | 有权限人员 | 草稿 | 生成主记录，status 默认"草稿" |
| 草稿 | 编辑状态为履行中 | 有权限人员 | 履行中 | update 修改 status |
| 履行中 | 到达到期日期 | 系统 | 已到期（提示） | 仅工作台/提醒展示到期，不自动改库中 status（BR-L03） |
| 履行中/已到期 | 编辑状态为已终止 | 有权限人员 | 已终止 | update 修改 status |
| 履行中/已到期/已终止 | 编辑状态为已归档 | 有权限人员 | 已归档 | update 修改 status |
| 任意状态 | 删除合同 | 有权限人员 | （已删除） | 级联删除关联数据，保留操作记录 |

**异常处理：**

| 异常场景 | 处理策略 | 错误反馈 |
|---------|---------|---------|
| 合同编号重复 | Service 前置 `findFirst` 校验 | 400 / 40001，`合同编号【XXX】已存在，请更换` |
| 日期顺序非法 | Service 前置校验 | 400 / 40002，"日期填写有误，请检查签订、生效、到期日期的先后顺序" |
| 更新/删除 id 不存在 | Prisma P2025 由全局过滤器兜底 | 400，"记录不存在" |
| 未登录/无权限 | 全局 Guard 依据 `@Perms` 元数据 | 401 / 403 |
| 参数格式非法 | class-validator DTO 管道 | 400，校验器首条 message |

## 3.2 合同详情 详细设计

**模块职责：** 单个合同的从属数据管理——附件归档、履约节点跟踪、收付款计划维护、操作记录只读展示。四类从属数据均通过 `contractId` 关联同一份 `ContractInfo` 主记录。不负责合同主记录增删改（属合同台账）、不负责提醒生成（属履约提醒定时任务）。（对应 PRD US-003/004/005、BR-CD01~CD11）

**核心类/服务设计（4 组独立 Controller/Service）：**

| 类/服务名 | 职责 | 关键方法 | 复用/自定义 |
|----------|------|---------|-----------|
| ContractAttachmentController / Service | 附件查询/上传/删除 | `list(contractId)`、`upload(dto,file)`、`delete(id)` | list/upload 自定义、delete 复用 |
| ContractMilestoneController / Service | 履约节点增删改/标记完成 | `list`、`add`、`update`、`complete`、`delete` | list/add/complete 自定义、update/delete 复用 |
| ContractPaymentPlanController / Service | 收付款计划增删改/登记 | `list`、`add`、`update`、`register`、`delete` | list/add/register 自定义、update/delete 复用 |
| ContractOperRecordController / Service | 操作记录只读 + 内部写入 | `list(contractId)`、`record(...)` | 不继承工厂（仅 list）；`record()` 供其他 Service 横向调用 |

> 合并策略：4 个业务域各自独立 Controller/Service（非合并进单一 ContractDetailController），因路由前缀已由 PRD 区分、读写模式差异大（附件涉文件上传、操作记录仅只读）、`OperRecordService.record()` 是"写操作记录"横切关注点。

**核心业务流程 — 附件上传：**

处理步骤：
1. 校验 contractId 对应合同存在，否则 40003「关联合同不存在」。
2. Multer 拦截器校验文件 ≤50MB（超出 40004）、category 命中枚举。
3. 校验失败或落盘异常 → 不创建附件记录，返回"上传失败，请重试"（BR-CD07），不进入写库。
4. 落盘成功后同一事务内：`contractAttachment.create()` 写元信息 + `OperRecordService.record(contractId,'附件上传',fileName,uploaderId)` 写操作记录（BR-CD10）。
5. 返回"新增成功"。

**核心业务流程 — 履约节点标记完成：**

处理步骤：
1. 校验节点 id 存在且非"已完成"（已完成再次调用幂等返回，不重复写记录）。
2. 事务内：更新 status='已完成'、写 actualDate/completeRemark（BR-CD03）→ 写操作记录'节点完成'（BR-CD10）。
3. 同步该节点关联的未处理履约提醒处理状态置"已处理"（跨模块，建议用领域事件解耦避免循环依赖）。
4. 返回"保存成功"。

**核心业务流程 — 收付款登记：**

处理步骤：
1. 校验计划 id 存在、actualAmount≥0。
2. 事务内：更新 status='已完成'、写 actualAmount/actualDate（BR-CD06，部分收付款按实际金额记录不拆分核销）→ 写操作记录'收付款完成'。
3. 返回"保存成功"。

**状态机（履约节点 / 收付款计划，两者流转一致）：**

| 当前状态 | 触发条件 | 触发方 | 流转至 | 系统动作 |
|---------|---------|-------|-------|---------|
| （无） | 新增节点/计划 | 有权限人员 | 待处理 | 默认状态（BR-CD01/CD04） |
| 待处理 | 计划日期已过且未完成/未登记（定时任务扫描） | 系统 | 逾期 | 批量置位（BR-CD02/CD05） |
| 待处理/逾期 | 标记完成/登记实际收付款 | 有权限人员 | 已完成 | 写实际数据 + 操作记录，不再判逾期（BR-CD03/CD06） |
| 已完成 | — | — | 已完成（终态） | — |

> 逾期判定由系统定时任务（复用现有 TaskInfo/TaskLog 机制新增 cron 任务）周期扫描 `status='待处理' AND planDate<今日` 批量置"逾期"，不在业务写路径实时计算。

**异常处理：**

| 异常场景 | 处理策略 | 错误反馈 |
|---------|---------|---------|
| 上传文件超 50MB | Multer limits.fileSize 拦截，不落盘不建记录 | 400 / 40004，"附件大小超过限制（最大50MB）" |
| 上传/落盘失败 | 捕获异常，不创建附件记录（BR-CD07） | 400，"上传失败，请重试" |
| 关联合同不存在 | Service 写前查 contractInfo，不存在则拒绝 | 400 / 40003，"关联合同不存在" |
| 附件分类枚举非法 | DTO `@IsIn` 拦截 | 400，"附件分类不合法" |
| 收付款方向/金额非法 | DTO 校验（direction 枚举、金额≥0） | 400，"参数不合法" |
| 访问已删合同的从属数据 | 外键级联保证一致性，查询返回空列表 | — |

## 3.3 履约提醒 详细设计

**模块职责：** 集中展示合同到期与履约节点相关站内提醒，支持查看来源、完成处理。提醒数据由系统定时扫描自动生成（派生数据），本模块提供跨合同的集中跟进视图。（对应 PRD US-006、BR-R01~R05）

**核心类/服务设计：**

| 类/服务名 | 职责 | 关键方法 | 复用/自定义 |
|----------|------|---------|-----------|
| ContractReminderController（继承 `CrudControllerFactory`，`prefix:'admin/contract/reminder'`） | 分页查询、查看来源、完成处理 | `list`（复用工厂分页）、`source(id)`、`handle(dto)` | list 复用、source/handle 自定义 |
| ContractReminderService（继承 `BaseService`，`super(prisma,'contractReminder')`） | 提醒查询、完成处理事务、被定时任务调用生成提醒 | `pageReminders`、`getSource`、`handleReminder`、`scanAndGenerate`（定时任务调用） | 自定义 |
| ContractReminderScanJob（复用 TaskInfo 机制的 cron） | 定时扫描合同到期日期、履约节点计划日期，生成/更新提醒 | `run()` | 自定义定时任务 |

**核心业务流程 — 完成处理：**

处理步骤：
1. 校验提醒 id 存在，读取其来源类型与来源 id。
2. 事务内：对来源履约节点执行标记完成（写 actualDate），同步该提醒 handleStatus='已处理'（BR-R03）。
3. 已处理提醒不再参与逾期状态判断。
4. 返回"处理成功"。

**核心业务流程 — 定时扫描生成提醒（scanAndGenerate）：**

处理步骤：
1. 定时任务周期触发（复用 TaskInfo/TaskLog）。
2. 扫描合同到期日期临近（默认前 30 天，可配）→ 生成/更新"合同到期提醒"。
3. 扫描履约节点计划日期临近 → 生成/更新"履约节点提醒"，handleStatus='待处理'（BR-R01）。
4. 履约节点超计划日期未完成 → 对应提醒 overdueStatus='已逾期'（BR-R02）。
5. 生成失败不影响合同主记录和履约节点维护（BR-R04），异常仅记日志。

**状态机（提醒处理状态）：**

| 当前状态 | 触发条件 | 触发方 | 流转至 | 系统动作 |
|---------|---------|-------|-------|---------|
| （无） | 计划日期临近 | 系统 | 待处理 | 生成或更新提醒 |
| 待处理 | 来源节点标记完成 / 完成处理 | 系统 / 有权限人员 | 已处理 | 不再参与逾期判断 |

**异常处理：**

| 异常场景 | 处理策略 | 错误反馈 |
|---------|---------|---------|
| 提醒 id 不存在 | Service 前置校验 | 400，"提醒不存在" |
| 来源合同/节点已删除 | 查询时过滤，不展示该提醒 | —（列表不返回） |
| 完成处理无权限 | 全局 Guard 依据 `@Perms('handle')` | 403 |
| 提醒生成失败 | 捕获异常记日志，不阻断业务 | —（不对外暴露） |

## 3.4 收付款管理 详细设计

**模块职责：** 跨合同集中查询、登记收付款计划并统计应收应付。不新增业务实体，复用合同详情的 `contract_payment_plan` 表，仅提供集中查询、登记入口和汇总统计。（对应 PRD US-007、BR-PM01~PM05）

**核心类/服务设计：**

| 类/服务名 | 职责 | 关键方法 | 复用/自定义 |
|----------|------|---------|-----------|
| ContractPaymentManageController（`prefix:'admin/contract/payment/manage'`） | 分页查询、登记、统计、导出 | `list`、`register`、`stat`、`export` | 全自定义（跨合同聚合） |
| ContractPaymentManageService（继承 `BaseService`，`super(prisma,'contractPaymentPlan')`） | 跨合同分页查询（join 合同名称/相对方）、登记事务、汇总统计 | `pagePlans`、`registerActual`、`statistics`、`listForExport` | 自定义 |

> 说明：登记实际收付款（register）与合同详情的 `payment/register` 为同一业务动作，Service 层可共享 `registerActual` 实现，两个 Controller 各自暴露入口。

**核心业务流程 — 收付款统计（stat）：**

处理步骤：
1. 按当前筛选条件（不含分页）聚合 contract_payment_plan。
2. 应收=收款方向计划金额合计；应付=付款方向计划金额合计。
3. 待收=收款方向 status='待处理' 金额合计；待付=付款方向 status='待处理' 金额合计。
4. 逾期=status='逾期' 金额合计（不分方向）。
5. 返回 5 项汇总，实时计算不落地。

**核心业务流程 — 登记实际收付款（register）：** 同 3.2 收付款登记流程（更新 status='已完成'、写实际金额/日期、写操作记录，事务内完成）。

**异常处理：**

| 异常场景 | 处理策略 | 错误反馈 |
|---------|---------|---------|
| 计划 id 不存在 | Service 前置校验 | 400，"收付款计划不存在" |
| actualAmount 为负 | DTO 校验 | 400，"实际金额不能为负数" |
| 来源合同已删除 | 查询时过滤 | —（列表不返回） |
| 登记/导出无权限 | 全局 Guard | 403 |

## 3.5 合同工作台 详细设计

**模块职责：** 登录后默认首页，实时汇总合同、履约节点、收付款计划、提醒数据，展示关键指标、待办提醒、看板图表、快捷入口和最近动态。不维护业务数据，不新增表。（对应 PRD US-001、BR-W01~W05）

**核心类/服务设计：**

| 类/服务名 | 职责 | 关键方法 | 复用/自定义 |
|----------|------|---------|-----------|
| ContractWorkbenchController（`prefix:'admin/contract/workbench'`） | 汇总数据、待办、最近动态查询 | `overview`、`todos`、`recent` | 全自定义（只读聚合） |
| ContractWorkbenchService | 跨表实时聚合统计 | `getOverview`、`getTodos`、`getRecent` | 自定义，依赖 contract/milestone/payment_plan/reminder |

**核心业务流程 — 工作台汇总（overview）：**

处理步骤：
1. 合同总数/履行中：按 contract.status 分组计数。
2. 即将到期：到期日期在临期阈值内（默认 30 天，可配）且状态未终止/归档的合同数。
3. 逾期合同：已过到期日期且状态未终止/归档的合同数（BR-W02 综合判定）。
4. 待收/待付：contract_payment_plan 按方向 + status='待处理' 金额合计；逾期金额：status='逾期' 合计。
5. 看板：状态分布/类型分布（contract 分组）、月度签约金额（按 signDate 月度聚合）、收付款完成情况（payment_plan 按 status 聚合）。
6. 无权限模块对应指标不返回（BR-W02 权限过滤）。所有数据实时计算不落地（BR-W04）。

**异常处理：**

| 异常场景 | 处理策略 | 错误反馈 |
|---------|---------|---------|
| 某区块无数据 | 返回空数组/0，前端显示"暂无数据" | —（正常返回） |
| 用户无某模块权限 | 该区块指标不计算不返回 | —（按权限裁剪） |
| 未登录 | 全局 Guard | 401 |

## 3.6 经营分析 详细设计

**模块职责：** 为管理者提供合同规模、金额分布、收付款进度和逾期风险的多维统计。纯统计展示，按筛选条件实时汇总 contract/milestone/payment_plan，不维护业务数据，不新增表。（对应 PRD US-008、BR-A01~A05）

**核心类/服务设计：**

| 类/服务名 | 职责 | 关键方法 | 复用/自定义 |
|----------|------|---------|-----------|
| ContractAnalysisController（`prefix:'admin/contract/analysis'`） | 经营分析汇总查询 | `overview` | 全自定义（只读聚合） |
| ContractAnalysisService | 按筛选条件多维聚合 | `getAnalysis(filter)` | 自定义 |

**核心业务流程 — 经营分析汇总（overview）：**

处理步骤：
1. 接收筛选条件：日期范围（按 signDate）、合同类型、合同状态，构建统一 where。
2. 指标：合同总数、合同总金额（amount 合计）、应收/应付/待收/待付/逾期（payment_plan 按方向与 status 聚合）。
3. 分布：按类型/状态对 contract 分组计数。
4. 趋势：按 signDate 月度聚合签约金额。
5. 收付款进度：payment_plan 按 status 聚合数量与金额；收付情况对比（应收/应付、已收/已付、待收/待付、逾期）。
6. 筛选条件变化后全部指标与图表同步刷新（BR-A01），实时计算不落地（BR-A04）。

| 异常场景 | 处理策略 | 错误反馈 |
|---------|---------|---------|
| 筛选下无数据 | 返回空/0，前端各区块显示"暂无数据" | —（正常返回） |
| 无经营分析权限 | 全局 Guard 依据菜单权限 | 403（菜单入口隐藏） |
| 未登录 | 全局 Guard | 401 |

# 五、API 详细设计

## 5.1 通用规范

- **响应格式**：统一 `ApiResponse<T>` / `PageResponse<T>`（见 2.3）。导出类接口返回 CSV 文本流（`Content-Type: text/csv`，UTF-8 BOM），不套用统一响应外壳（沿用现有 user 导出惯例）。
- **鉴权**：所有接口需登录（JWT）。写操作标注权限码 `contract:<资源>:<操作>`，由 `@Perms` + 前缀派生（`admin/contract/ledger` → `contract:ledger:*`）。
- **列表查询约定**：合同台账、履约提醒、收付款管理为跨记录分页查询（`GET .../list?query`）；合同详情的从属数据为按合同查询（`GET .../list/:contractId`，返回该合同全量列表，不分页），后者需覆盖 CRUD 基类默认的 query 分页实现。
- **事务约定**：涉及多表写入（上传+操作记录、完成+提醒同步、删除级联）的接口在 `prisma.$transaction` 内完成。

## 5.3 接口详细设计（代表性自定义接口）

### 5.3.1 新增合同

- **POST** admin/contract/ledger/add，权限码 `contract:ledger:add`
- **请求体（AddContractDto）**：code(必填,唯一)、name、type(枚举)、signSubject、counterparty、amount(≥0,2位小数)、currency(枚举,默认人民币)、signDate、effectiveDate、expireDate、ownerId(必填)、status(枚举,默认草稿)、remark(可选)
- **成功响应**：`{ "code": 200, "msg": "新增成功", "data": {ContractVo} }`
- **错误码**：40001 合同编号已存在；40002 日期顺序非法；400 参数校验失败
- **事务/幂等**：单表写入无需事务；非幂等，前端 loading 防重复提交

### 5.3.2 删除合同（级联）

- **DELETE** admin/contract/ledger/delete/:id，权限码 `contract:ledger:delete`
- **成功响应**：`{ "code": 200, "msg": "删除成功" }`
- **事务**：`$transaction` 内删附件→节点→计划→提醒→主记录；操作记录保留
- **错误码**：400 记录不存在

### 5.3.3 上传合同附件

- **POST** admin/contract/attachment/upload（multipart/form-data），权限码 `contract:attachment:upload`
- **请求参数**：contractId(必填,须存在)、category(必填,枚举)、file(必填,≤50MB)
- **成功响应**：`{ "code": 200, "msg": "新增成功", "data": {ContractAttachmentVo} }`
- **错误码**：40003 关联合同不存在；40004 附件超 50MB；400 分类非法/上传失败（不建记录）
- **事务**：落盘成功后同事务写附件记录 + 操作记录；落盘失败短路不写库

### 5.3.4 标记履约节点完成

- **PUT** admin/contract/milestone/complete，权限码 `contract:milestone:complete`
- **请求体**：id(必填,须存在且非已完成)、actualDate(必填)、completeRemark(可选,≤200)
- **成功响应**：`{ "code": 200, "msg": "保存成功" }`
- **事务**：更新节点状态 + 写操作记录 + 同步关联提醒为已处理，同事务完成

### 5.3.5 登记实际收付款

- **PUT** admin/contract/payment/register（及 payment/manage/register），权限码 `contract:payment:register`
- **请求体**：id(必填,须存在)、actualAmount(必填,≥0,2位小数)、actualDate(必填)
- **成功响应**：`{ "code": 200, "msg": "保存成功" }`
- **事务**：更新计划状态='已完成' + 写操作记录，同事务完成

### 5.3.6 收付款统计

- **GET** admin/contract/payment/manage/stat，权限码 `contract:payment:manage:stat`
- **请求参数**：同分页查询（不含分页）
- **成功响应**：`{ "code": 200, "data": { receivableTotal, payableTotal, pendingReceive, pendingPay, overdueTotal } }`
- **说明**：实时聚合，不落地

> 其余复用工厂接口（list/detail/update/delete）遵循 `CrudControllerFactory` 标准行为与统一响应；从属数据 list 接口为 `/:contractId` 路径参数、返回全量列表。工作台/经营分析各聚合接口的响应结构见 PRD 对应章节。

# 六、安全设计

| 维度 | 设计 |
|------|------|
| 身份认证 | 所有接口经 JWT 鉴权，未登录返回 401，复用系统设置模块认证体系 |
| 权限控制 | 写操作（增删改、上传、登记、导出）标注 `@Perms('contract:*')`，无权限 403；查询类仅需登录（合同数据对所有登录用户可见） |
| 越权防护 | 完成处理、登记、删除等受控操作按角色权限控制入口，前端无权限隐藏、后端强制校验 |
| 输入校验 | 所有入参经 class-validator DTO 校验（类型/必填/长度/枚举/数值范围）；外部数据使用前校验 |
| 文件上传安全 | 附件限制单文件 ≤50MB（Multer limits）；校验分类枚举；落盘失败不建记录 |
| 数据一致性 | 多表写入用事务保证原子性；删除合同级联清理关联数据，操作记录保留审计 |
| 注入防护 | 统一经 Prisma 参数化查询，无裸 SQL 拼接 |
| 敏感操作审计 | 附件上传、节点完成、收付款完成等关键操作写入 contract_oper_record，不可篡改（无编辑/删除入口） |
| 金额安全 | 金额仅记录与统计，不涉及核销、发票匹配、汇率换算，降低资金操作风险面 |
